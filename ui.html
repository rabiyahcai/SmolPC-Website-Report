UI Design

Design Principles

The UI was designed around a single guiding idea: users should not have to learn a different application shell for each assistant. Instead of building four disconnected interfaces, SmolPC uses one shared desktop shell and changes the active mode, help content, setup state, and prompt suggestions depending on the task. This supports the broader project goal of a unified offline platform rather than a collection of unrelated demos.

Simplicity

Simplicity was applied by keeping the product inside one persistent shell with one main input pattern. The user always sees the same sidebar, conversation area, composer, and workspace header, while more advanced information such as hardware details, model status, and setup troubleshooting is pushed into overlays or drawers instead of crowding the main flow. This means that switching from Code to GIMP or Writer changes the task context without forcing the user to relearn the whole interface.

Consistency

Code, LibreOffice, GIMP, and Blender all use the same core layout: shared sidebar, common header controls, the same conversation view, the same composer bar, and the same setup and status language. This means that switching tools feels like changing context inside one product rather than launching a new product each time.

Visibility

Important state is always visible. The interface exposes the active mode, setup readiness, model and runtime information, loading states, and host-application availability so that users can understand whether the assistant is ready and what environment it is currently using.

Feedback

Feedback is provided through startup loading screens, setup banners, streaming assistant responses, quick examples, availability messages, and error surfaces for host-app launch or inference issues. This is particularly important for a local AI system, where users need to understand why a request may be slower or blocked on a prerequisite.

Tolerance

The interface is designed to help users recover rather than fail abruptly. Setup gating explains missing prerequisites, mode help panels explain what a mode is for, and chat history persists across sessions so users do not lose their place when a workflow needs to be retried.

Learnability

Each mode uses short, mode-specific welcome copy and example prompts to teach the user what the assistant can do. This matters because the same shell supports very different tasks, from coding help to image editing and Blender guidance.

Unified App Shell

The implemented interface is centred on a reusable application shell. This shell gives every mode the same basic structure while still allowing mode-specific behaviour, copy, setup checks, and host-application controls. The result is a UI that feels coherent even though the underlying assistants target very different workflows.

UI Element Purpose Why It Matters
Sidebar Shows and switches between saved conversations. Supports continuity and makes the app feel like a persistent workspace rather than a one-off prompt box.
Mode selector Changes the active assistant workflow without leaving the main shell. Reinforces the idea of one platform with multiple specialised modes.
Workspace header Shows current context and provides access to help, model info, hardware, and diagnostics. Keeps important system state visible without cluttering the main conversation area.
Startup loading screen Explains that the app is checking setup and waiting for engine readiness before normal use. Prevents the interface from appearing broken while local runtime preparation is still in progress.
Setup banner and setup panel Warns the user when a mode is unavailable or missing prerequisites. Essential for host-integrated tools such as GIMP, Blender, and LibreOffice.
Conversation view and welcome state Displays mode-specific onboarding copy, example prompts, and the running conversation. Supports both first-time guidance and ongoing use in the same space.
Composer bar Provides a single input pattern for all text-based interactions, including an optional microphone control in SmolPC 2.0's shared chat shell. Reduces cognitive load by keeping the core interaction model consistent while still allowing low-friction dictation where supported.
Per-message voice controls Allow the user to play or stop local read-aloud on assistant responses. Supports accessibility and hands-free use without sending content to a cloud TTS service.
Model recommendation banner Warns when a tool-backed mode is likely to perform badly on the currently selected small model. Helps users make a sensible performance-quality tradeoff without exposing raw backend details first.
Help drawer and overlays Explain mode purpose, controls, troubleshooting steps, and runtime status. Improves learnability and gives non-expert users confidence in an unfamiliar local AI workflow.
Host app launch controls Open or focus the relevant external application from the current workflow. Bridges the gap between the assistant shell and real user tasks inside GIMP, Blender, or LibreOffice.

Code Mode

Code mode is the clearest and simplest mode in the current UI. It uses the shared shell without needing an external host application, which makes it the easiest place to establish the product's overall interaction pattern.

Implemented UI Decisions

This mode emphasises a clean conversation view, clear prompt starters, visible runtime status, and low-friction onboarding. The welcome state explains what Code mode is for, the shared conversation view streams responses progressively, and the same shell exposes keyboard shortcuts, model information, and hardware status when needed. Code mode is also the clearest example of how optional voice controls work inside SmolPC 2.0: a microphone button in the composer for local dictation and a read-aloud control on assistant messages for local playback.

LibreOffice Modes

The LibreOffice workflows reuse the same shell as Code mode but place more emphasis on setup readiness, host-application launching, and concise structured task descriptions. This lets the UI support document and presentation work without inventing a totally separate interaction model.

Implemented UI Decisions

Writer and Slides use mode-specific welcome copy, a shared LibreOffice launch control, and setup messaging that makes it clear the user is working with a real desktop tool rather than a generic chatbot. The UI also warns when a stronger tool-capable model would be more suitable, which helps these modes stay understandable on lower-spec machines.

High-Fidelity Wireframe

GIMP Assistant

GIMP mode required a stronger emphasis on host-state awareness than Code mode because the usefulness of the assistant depends on the external editor being available and correctly prepared. The UI therefore gives setup, mode availability, and edit feedback a more prominent role.

Implemented UI Decisions

The mode-specific copy encourages direct plain-English image-editing prompts, while setup surfaces explain what needs to be ready before the assistant can act. The shared shell then reuses the same conversation view and launch controls, which keeps the workflow approachable for users who may not know GIMP's menu structure well but still need clear recovery guidance if the bridge or host app is unavailable.

High-Fidelity Wireframe

Blender Assistant

Blender mode has the most context-heavy UI story because it is designed around scene-aware guidance rather than generic chat alone. The interface therefore uses its welcome copy and help surfaces to steer users toward asking about the live scene, objects, modifiers, materials, and specific modelling steps.

Implemented UI Decisions

Blender mode keeps the shared shell but places extra importance on clear troubleshooting, contextual guidance, and mode help. This is because the user is often moving between the assistant and a live 3D environment, so the design needs to keep scene-aware guidance discoverable without overwhelming the main conversation area.
