System Design

System Architecture

The final system is designed as a unified desktop application rather than a set of unrelated assistants. The current integrated app is CodeHelper, which acts as the shared shell for multiple modes. Behind that shell sits a shared SmolPC inference engine, a mode-provider layer that routes requests to the correct assistant logic, and a set of host-application bridges for tools such as GIMP, Blender, and LibreOffice. This separation is one of the most important design decisions in the project because it allows inference, UI, and host-specific integrations to evolve independently.

High-Level Architecture

The high-level diagram for the final report should show five layers: the user-facing unified desktop UI, the Tauri command boundary, the mode-provider layer, the shared SmolPC engine, and the external host applications. The key point is that inference ownership belongs to the shared engine, while workflow-specific orchestration belongs to the app and its providers.

Core Components

Unified App Shell

The current integrated product is implemented in apps/codehelper. Its Svelte frontend provides the conversation UI, mode switching, setup guidance, hardware and model panels, mode-specific help, and lightweight local persistence for user-facing state such as chats and settings. The same shell presents CodeHelper, GIMP, Blender, and LibreOffice workflows such as Writer and Slides through one interface.

Shared SmolPC Engine

The shared engine is split into smolpc-engine-core, smolpc-engine-host, and smolpc-engine-client. It owns startup lifecycle, readiness, backend and model selection, generation, cancellation, and the localhost API contract. This keeps model/runtime ownership out of individual assistants and makes it possible to reuse the same local runtime across multiple modes.

Mode Provider Layer

The backend uses a registry of tool providers to map each mode to its corresponding orchestration logic. Code uses a local provider, GIMP uses a dedicated provider, Blender uses a scene-aware provider, and the LibreOffice workflows share one office provider family. This design keeps the frontend unaware of host-specific implementation details and makes the mode system extensible.

Host Application Bridges

External applications are not controlled directly by the chat UI. Instead, each workflow uses an application-specific bridge. GIMP uses a bundled MCP bridge, Blender uses a local bridge and addon for scene-aware access, and LibreOffice uses an MCP-backed runtime and helper bridge. These bridges allow the assistant layer to call structured tools rather than relying on unstructured text only.

Launcher and Setup Layer

The current branch also contains launcher-oriented orchestration and manifest support. At the moment this is still transitional rather than a fully separate standalone launcher application, but it already supports manifest-driven launch or focus behaviour and contributes to the longer-term single-suite packaging direction. The setup layer also checks runtime resources and host-app availability so the UI can gate modes when prerequisites are missing.

Interaction Flows

The final diagrams for this page should focus on the main architectural flows rather than repeating very similar assistant-specific pictures. Two sequence diagrams are especially important: a pure text-generation flow and a host-assisted tool-execution flow.

Text Generation Flow

In Code mode, the user writes a request in the Svelte frontend. The frontend sends this through a Tauri command to the backend, which ensures the engine is running and ready. The backend then calls the shared engine client, which communicates with the localhost engine host. Tokens are streamed back to the frontend and appended to the active chat. This flow demonstrates the clean boundary between UI, Tauri backend, and shared engine.

Host-Assisted Mode Flow

In GIMP, Blender, and LibreOffice modes, the frontend still sends one assistant request, but the backend first resolves the correct provider from the mode registry. The provider then checks the relevant bridge, gathers tool status or context, and may use the shared engine to plan or phrase the response before executing structured host actions. This is the key flow that distinguishes the unified assistant platform from a plain local chatbot.

Availability and Launch Flow

On startup, the app checks runtime resources, engine readiness, and host application availability. These checks drive the setup UI and the mode availability state shown to the user. When supported, host applications can be launched or focused through command-based orchestration rather than requiring the user to manage every dependency manually.

Data Storage and State

This project does not centre around a relational database, so a traditional ER diagram is not the most useful design artefact. The more relevant design question is where lightweight user state, runtime resources, and shared engine assets are stored.

Design Patterns

The most important design patterns in this project are architectural rather than purely object-oriented. The system is designed around strong boundaries between inference ownership, application orchestration, and host-specific tooling.

Contract-First Boundary

Applications talk to the engine through the typed engine client or the documented localhost HTTP API instead of depending on engine internals. This makes the engine reusable across products and reduces the chance that one assistant becomes tightly coupled to one runtime implementation.

Provider Registry / Strategy Pattern

The mode registry chooses the correct provider family for each mode. This is effectively a strategy-style design: the frontend sends a unified request, and the backend delegates behaviour to the provider that knows how to handle that mode’s tools, context, and bridge logic.

Command Boundary Through Tauri

The Tauri command layer acts as a stable boundary between the Svelte frontend and the Rust backend. This keeps the UI code focused on user interaction while backend commands handle engine startup, assistant execution, mode status, setup preparation, and launch orchestration.

Applicability of Template Items

Some standard report-template items are less central to this project than the architecture and API boundaries above. A traditional website sitemap is not the best fit because the product is a single-window desktop app with mode switching rather than a multi-page web application. Likewise, a relational ER diagram is not central because the system does not use a database as a core architectural component. If time is limited, the final report should prioritise the high-level architecture diagram, one or two strong sequence diagrams, provider boundaries, and API contracts over generic placeholder diagrams.
