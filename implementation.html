Implementation

Overview

The implementation moved from separate prototypes toward a unified desktop product built around one shared engine and one integrated application shell. The strongest implementation theme is therefore not four isolated assistants, but the interaction between the shared engine, the Tauri backend, the Svelte frontend, and the mode-specific providers that adapt the platform to different workflows.

Shared Engine Integration

The shared engine is responsible for startup, readiness, backend selection, model loading, generation, and cancellation. The frontend does not talk to inference libraries directly. Instead, the Svelte app uses Tauri commands, and those commands delegate to the shared engine client. This design removes per-assistant runtime ownership and makes the local inference path reusable across multiple modes.

Runtime and Model Selection

The backend supports multiple local runtime lanes and resolves the best supported path for the current machine. This is important because the product is intended for offline local use across varying hardware rather than one fixed benchmark device. The implementation therefore includes startup/readiness handling, runtime mode selection, model registry support, and lane-specific status reporting.

Communication Between Components

The communication path is layered. The user interacts with the Svelte frontend, which calls a Tauri command. The Tauri backend either routes the request to the shared engine or to a mode-specific provider. The provider may then call the engine, a host-app bridge, or both. This is the key implementation pattern that ties together the unified product.

Unified App Shell

The current integrated application lives in apps/codehelper. Although Code mode is the flagship workflow, the SmolPC 2.0 app shell now supports multiple workflows in one interface, including Code, GIMP, Blender, and LibreOffice modes such as Writer and Slides.

Mode Configuration and Switching

Modes are configured as structured frontend objects with labels, capability flags, suggestions, and provider types. The UI does not hard-code one assistant flow; instead, it loads mode metadata and presents the appropriate controls for the selected mode. This was an important step in turning a single assistant UI into a platform shell.

Streaming Chat Flow

Assistant requests are sent through a unified frontend API using Tauri channels for streaming events. This allows the interface to display partial tokens, completion, cancellation, and assistant-specific error events in one consistent conversation view. The streaming behaviour is especially important for making local generation feel responsive even when running heavier models.

Local Persistence

The frontend stores chats, current mode chat IDs, and settings in localStorage. Chats are organised per mode rather than as one global list, which allows the user to switch between workflows without losing context. Settings such as runtime preference, selected model, and theme are also persisted locally.

Setup and Availability Gating

The UI does not assume that every host application is installed or ready. Instead, startup and setup checks expose whether GIMP, Blender, or LibreOffice are available, and the frontend maps those results into mode availability. This implementation detail is important because the platform is designed to operate across different machines with different installed applications.

Assistant Routing and Providers

A major implementation feature is the provider registry in the Rust backend. The assistant request layer receives a unified assistant request and then dispatches it to the correct provider family based on the selected mode. This keeps the UI simple while still allowing each mode to have different execution behaviour.

Provider Family Modes Implementation Role
Code provider Code Direct text generation through the shared engine
GIMP provider GIMP Bridge management, tool discovery, tool execution, and undo support
Blender provider Blender Scene bridge management, local RAG retrieval, and scene-aware tool support
LibreOffice provider Writer, Slides, and related office workflows Shared office-mode runtime handling, tool filtering, and office workflow orchestration

Code Mode and Core Chat Experience

Code mode is the simplest implementation path and the foundation of the unified app shell. It uses the shared engine for local text generation and presents the result through the same conversation UI that the other modes reuse. It is also currently one of the clearest places where the platform's voice features are surfaced to the user.

Key Implementation Features

Blocking engine startup handshake: the inference store ensures that the engine is started and ready before normal generation begins, and then refreshes backend status so the UI can show the active runtime state.

Streaming generation: responses are streamed token by token through the Tauri command boundary, which allows the frontend to update the active assistant message incrementally.

Memory and runtime awareness: the inference layer also exposes runtime mode, backend status, and memory-pressure information so the UI can react to constrained devices more gracefully.

Voice Input and Read-Aloud

Voice features were implemented as an extension of the existing SmolPC 2.0 chat workflow rather than as a separate subsystem from the user's perspective. In the current unified app shell, these features are exposed through an optional microphone button in the composer and a read-aloud control on assistant responses.

Frameworks and Libraries

Framework / Library Purpose How It Was Used
cpal Microphone capture Used in the Tauri layer to record audio from the user's input device at its native sample rate
rubato Audio resampling Used to convert recorded audio into the 16kHz mono format expected by the local Whisper transcription path
Shared engine + WhisperPipeline Local speech-to-text Used to transcribe microphone audio through the engine's /v1/audio/transcriptions endpoint
rodio Audio playback Used to play returned WAV data locally when the user requests read-aloud on an assistant message
smolpc-tts-server and KittenTTS Local text-to-speech Used as a detached sidecar behind the engine's /v1/audio/speech proxy so that text-to-speech stays offline without complicating the main inference process

Key Features Implementation

Microphone capture and transcription insertion: the app records locally, mixes and resamples the audio where needed, sends it to the engine, and inserts the resulting text back into the composer so the user can review it before sending.

Per-message read-aloud: assistant responses can be synthesized on demand and played back locally, with the UI tracking loading, playing, and stop states through a dedicated voice store.

Detached TTS boundary: text-to-speech is intentionally isolated behind a sidecar process and the engine API so that voice-output problems degrade to an unavailable control instead of destabilising the main chat or runtime-selection path.

GIMP Mode

GIMP mode is implemented as a host-assisted workflow rather than plain text chat. The provider is responsible for preparing the required runtime, detecting GIMP, establishing the MCP session, discovering tools, and executing image-editing actions.

Frameworks and Libraries

Framework / Library Purpose How It Was Used
Tauri + Rust Desktop app backend and process orchestration Used to manage the GIMP bridge runtime, tool execution, and setup flow
MCP client layer Structured tool communication Used to connect to the GIMP MCP bridge, list tools, and execute actions
Shared engine Local text generation Used when natural-language requests need planning or text generation support

Key Features Implementation

Bundled runtime preparation: the provider prepares or validates the required GIMP-side runtime resources before trying to connect, which reduces manual setup for the user.

Connection and tool discovery: the provider connects to the GIMP bridge, stores discovered tools, and exposes mode state back to the frontend.

Undo support: GIMP mode is one of the few modes in the unified app that explicitly exposes undo through the provider interface.

Blender Mode

Blender mode combines a host bridge with local context retrieval. This makes it different from both Code mode and GIMP mode: it is not just a tool executor, and it is not just a plain text assistant. Instead, it is designed to answer scene-aware questions and support Blender workflows using both bridge state and local retrieval assets.

Frameworks and Libraries

Framework / Library Purpose How It Was Used
Tauri + Rust Backend orchestration and desktop integration Used to manage the Blender scene bridge and assistant routing
Bridge runtime and addon Scene-aware host communication Used to retrieve live scene information and expose Blender-specific tooling
Local RAG index Reference retrieval Used to load local Blender reference documents and enrich responses with relevant context

Key Features Implementation

Scene bridge startup: the provider starts or connects to a local scene bridge and stores runtime state so Blender mode can remain scene-aware across requests.

RAG-backed assistance: the provider loads a local retrieval index and combines it with live scene information, allowing responses to use both static reference material and current workspace context.

Hybrid execution path: Blender mode can gather scene state, retrieve reference context, and use the shared engine for generated explanations within one request flow.

LibreOffice Modes

The LibreOffice workflows are implemented through one shared provider family. This reflects the system design decision that office features should share runtime management and tool orchestration instead of being implemented as completely separate products.

Frameworks and Libraries

Framework / Library Purpose How It Was Used
Tauri + Rust Backend orchestration Used to manage LibreOffice runtime setup, provider state, and assistant execution
MCP-backed office runtime Structured office tool access Used to connect Writer and Slides workflows to office actions through an allowlisted tool layer
Shared engine planner and streamer Planning and summarisation Used to decide whether to call a tool and to produce short user-facing summaries after a successful office action

Key Features Implementation

Mode-specific tool filtering: the provider filters the raw office tool list based on the selected mode so that each office workflow only exposes the tools appropriate to that task.

Planner then tool execution: the office executor first uses the engine to decide whether a tool call is needed, then executes the chosen office action, and finally generates a short summary for the user.

Shared provider family: the office workflows are grouped under one provider family, which reduces duplication and fits the unified product architecture.

Launcher and Setup Features

The project also includes implementation work for setup preparation and launch orchestration. These pieces are important because the product goal is not only local AI, but local AI that is practical to run.

Setup commands: the backend exposes commands for reading setup status and preparing local resources, which allows the frontend to guide the user when runtimes or host apps are missing.

Mode open/focus behaviour: the backend can detect and launch host applications such as GIMP, Blender, and LibreOffice, making the unified shell more than a passive chat window.

Manifest-driven launcher direction: launcher modules and manifest handling are already present in the codebase, supporting the longer-term goal of a cohesive suite and easier deployment.
