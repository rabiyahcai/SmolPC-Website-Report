Project Background

Artificial intelligence is transforming education, yet access to AI-powered learning tools remains deeply unequal. Leading solutions such as Microsoft Copilot and Google Gemini require paid subscriptions and reliable internet connectivity, placing them out of reach for the millions of students attending under-resourced schools worldwide. According to UNESCO, the majority of primary schools and a significant proportion of secondary schools worldwide lack internet connectivity. [1] Meanwhile, teachers in these schools continue to spend excessive time manually producing lesson materials with little technological support.Full-time teachers already work over 50 hours per week [2], leaving little time to manually produce the diverse learning materials that modern classrooms demand. The result is a widening attainment gap between students in well-funded institutions and those without access to modern EdTech tools. [3]

Existing open-source alternatives offer some promise but fall short in practice. The original SmolPC LibreOffice Assistant demonstrates that locally-run AI is technically feasible, yet remains limited to a single application and lacks the educational features needed for real classroom use. Broader frameworks such as OpenVino have matured significantly, making it increasingly practical to run capable language models on affordable consumer hardware, but no existing solution brings these capabilities together into a cohesive, educationally-focused suite that works out of the box for students and teachers.

It was against this backdrop that SmolPC 2.0 was conceived, under the supervision of Professor Dean Mohamedally at UCL's Department of Computer Science. Building upon the original SmolPC framework, this project extends the platform into a multi-application AI assistant suite spanning LibreOffice, CodeHelper, GIMP, and Blender, with the goal of making AI-assisted learning freely accessible to every student, regardless of their school's budget or internet connectivity.

Project Goals

The overarching goal of SmolPC 2.0 is to deliver a free, offline, open-source AI assistant suite that integrates directly into the software students already use: LibreOffice, CodeHelper, GIMP, and Blender, and runs on low-cost hardware such as Intel N200 laptops costing as little as £300.

Specifically, SmolPC 2.0 aims to:

  • Enable students and teachers to control educational software through natural language, reducing the technical barrier to using complex applications.
  • Support coding education through an AI-powered assistant that guides students through programming concepts and helps them debug their work.
  • Teach image editing and 3D modelling interactively, through AI assistants embedded directly in GIMP and Blender.
  • Operate entirely offline using locally deployed large language models via OpenVino, ensuring data privacy and removing the dependency on internet connectivity.
  • Run efficiently on low-spec, affordable hardware to ensure accessibility for under-resourced schools.

    • Gathering Requirements

    Because SmolPC v2.0 is an extension of an existing project, we began by reviewing the original SmolPC v1.5 specification and meeting with our supervisor, Prof. Dean Mohamedally, who acts as our project partner. Through these discussions, we established the core direction of the project:

  • The tool must run entirely offline on budget school laptops (under £300), such as Intel N200 devices, with a minimum of 4 GB RAM.
  • Data privacy is non-negotiable. No student data can leave the device, ruling out cloud-based AI solutions entirely.
  • The system must be free and open-source, as schools cannot afford subscription-based tools.
  • The primary tools to build are an AI assistant for LibreOffice (Writer and Impress), an AI coding tutor (CodeHelper), and learning assistants for GIMP and Blender.

    • We also conducted a comparative analysis of cloud AI tools (Microsoft Copilot, Google Gemini, ChatGPT) against local AI alternatives to validate this direction. The analysis confirmed that cloud solutions are unsuitable for UK state schools due to GDPR constraints, cost barriers, and dependence on stable internet connectivity. This reinforced the need for a fully offline, local AI platform.

    Use Cases

    The following use cases describe how students and teachers interact with SmolPC v2.0. Each use case maps to one or more functional requirements listed in the MoSCoW section below.

    UC1 — Send a Voice or Text Command to the AI

    ActorStudent / Teacher
    Pre-conditionSmolPC 2.0 Home is open and a local AI model is running via Ollama.
    Main Flow
    1. User opens SmolPC 2.0 Home and selects a tool.
    2. User types a request in the chat box, or presses the microphone button to speak.
    3. If voice, Whisper transcribes the audio to text locally.
    4. The text is sent to the local LLM; a loading indicator is shown while processing.
    5. The AI response appears in the chat panel.
    Post-conditionThe AI response is visible in the chat and, where applicable, applied to the open document.
    ExceptionIf Ollama is not running, a clear error message is shown with a restart prompt.

    UC2 — Create or Edit a Document in LibreOffice Writer

    ActorStudent / Teacher
    Pre-conditionLibreOffice ≥ 7.6 is installed and the SmolPC 2.0 extension is loaded.
    Main Flow
    1. User opens LibreOffice AI from SmolPC 2.0 Home.
    2. User types or speaks a command (e.g. "Write an introduction about photosynthesis" or "Reformat this as bullet points").
    3. The AI reads the current document context via the LibreOffice UNO API.
    4. Generated content is previewed in the chat before being inserted into the document.
    5. User confirms; the document is updated. Undo/redo remains available.
    Post-conditionThe document is updated as instructed. The action can be undone.

    UC3 — Create a Presentation in LibreOffice Impress

    ActorTeacher
    Pre-conditionLibreOffice AI extension is active; Impress is installed.
    Main Flow
    1. Teacher types a command such as "Create a 5-slide presentation on the water cycle for Year 8."
    2. The AI generates slide titles, bullet points, and speaker notes via the UNO API.
    3. Accessible formatting is applied automatically (high-contrast colours, font size ≥ 18pt).
    4. Teacher reviews the slides inside Impress and requests edits via natural language.
    Post-conditionA complete, accessible presentation is saved to disk. No internet was required.

    UC4 — Get Coding Help from CodeHelper

    ActorStudent
    Pre-conditionSmolPC 2.0 CodeHelper is open; a code-focused model (e.g. Llama 3.2 or DeepSeek) is loaded.
    Main Flow
    1. Student types a coding question or pastes broken code into the chat.
    2. Student optionally selects a quick-example chip (e.g. "loops", "functions", "file I/O").
    3. CodeHelper returns working code with inline comments in plain English.
    4. Student copies the code or asks a follow-up question; chat history persists.
    Post-conditionStudent receives correct, well-commented code with a plain-English explanation suited to their age group.

    UC5 — Learn GIMP or Blender with Step-by-Step Guidance

    ActorStudent
    Pre-conditionGIMP or Blender is installed on the device; the corresponding SmolPC 2.0 assistant is selected.
    Main Flow
    1. Student asks how to do something (e.g. "How do I remove the background from an image in GIMP?").
    2. The assistant provides numbered, step-by-step instructions in plain English.
    3. Student follows the instructions in GIMP or Blender and asks follow-up questions as needed.
    4. The assistant clarifies any steps or suggests alternatives if the student gets stuck.
    Post-conditionStudent completes the task themselves having learned the relevant workflow, rather than having it done for them.

    UC6 — Switch Between Tools

    ActorStudent / Teacher
    Pre-conditionUser is inside any SmolPC 2.0 tool.
    Main Flow
    1. User opens the tool dropdown in the sidebar.
    2. User selects a different tool (e.g. switches from LibreOffice AI to CodeHelper).
    3. The interface updates without closing the application. Chat history is preserved per tool.
    4. Optionally, user selects a different local model from the model selector.
    Post-conditionUser is in the selected tool with no data loss; previous chats remain accessible in the sidebar.

    UC7 — View and Continue a Previous Chat Session

    ActorStudent / Teacher
    Pre-conditionAt least one previous session exists locally.
    Main Flow
    1. User opens SmolPC 2.0 and sees their chat history listed in the left sidebar.
    2. User clicks a past conversation (e.g. "Python Function Help — Yesterday").
    3. The full conversation loads and the user continues from where they left off.
    Post-conditionUser resumes a previous session with all context retained.

    MoSCoW Requirements

    Requirements were prioritised using the MoSCoW method, separating what the system must deliver from what would be beneficial but is not critical. The list below reflects the final agreed requirements following stakeholder interviews and pseudo-user testing.

    Functional Requirements

    ID Requirement Priority
    FR-M1 The system shall accept natural language commands via text input across all tools. Must Have
    FR-M2 The system shall accept voice commands and transcribe them locally using a Whisper model, with no audio sent externally. Must Have
    FR-M3 The system shall allow users to create, edit, and format documents in LibreOffice Writer via AI commands. Must Have
    FR-M4 The system shall allow users to create and edit presentations in LibreOffice Impress via AI commands. Must Have
    FR-M5 All AI inference shall run entirely on-device with no internet connection required after initial setup. Must Have
    FR-M6 The system shall run on devices with a minimum of 4 GB RAM, including Intel N200 and ARM-based laptops under £300. Must Have
    FR-M7 CodeHelper shall answer coding questions, explain programming concepts, and generate working, commented code suited to students aged 11–18. Must Have
    FR-M8 SmolPC 2.0 Home shall provide a unified launcher for all tools without requiring the user to open separate applications. Must Have
    FR-S1 The system shall display a visible loading indicator while the AI model is processing a request. Should Have
    FR-S2 The system shall persist chat history per tool across sessions so users can continue previous conversations. Should Have
    FR-S3 The GIMP and Blender assistants shall provide step-by-step instructional guidance, teaching users how to use the software themselves rather than controlling it directly. Should Have
    FR-S4 Generated content shall be filtered for school-appropriate language before being displayed to the user. Should Have
    FR-S5 The system shall support at least 3 languages for input and output. Should Have
    FR-S6 The system shall leverage Intel NPU hardware acceleration where available, with transparent fallback to CPU-only inference. Should Have
    FR-C1 A teacher dashboard could display student usage patterns and session frequency, stored entirely on the local device. Could Have
    FR-C2 A RAG-based help system could retrieve relevant LibreOffice, GIMP, or Blender documentation in response to user queries. Could Have
    FR-C3 The UI could offer a high-contrast mode and dyslexia-friendly font options to further support users with accessibility needs. Could Have
    FR-W1 The system will not use cloud-hosted AI models or transmit any user data to external servers. Won't Have
    FR-W2 The system will not require a paid subscription or API key for any core functionality. Won't Have
    FR-W3 Full school network deployment or MDM integration is out of scope for this project phase. Won't Have

    Non-Functional Requirements

    ID Requirement Category Priority
    NF-M1 All AI functions shall work fully offline after initial model download — no network calls during normal use. Privacy / Security Must Have
    NF-M2 No student data, prompts, or generated content shall leave the local device at any time. Privacy / GDPR Must Have
    NF-M3 The system shall run without crashing or becoming unresponsive on a device with 4 GB RAM and an Intel N200 processor. Performance Must Have
    NF-M4 All source code and AI models used shall be open-source and redistributable under compatible licences. Legal / Licensing Must Have
    NF-S1 Users shall be able to complete common tasks in no more than 3 interface steps. Usability Should Have
    NF-S2 Voice input and screen-reader compatibility shall be available across all primary tools. Accessibility Should Have
    NF-S3 AI responses shall be generated in under 30 seconds on CPU-only hardware and under 5 seconds with Intel NPU acceleration. Performance Should Have
    NF-S4 The architecture shall be extensible — adding a new tool shall not require changes to the core SmolPC 2.0 codebase. Maintainability Should Have
    NF-S5 Installation and setup shall complete in under 15 minutes on a fresh device with no prior configuration. Deployability Should Have
    NF-C1 The UI could support a high-contrast mode and dyslexia-friendly font options. Accessibility Could Have
    NF-C2 The system could log anonymous performance metrics locally (e.g. response time, model used) to assist future optimisation. Maintainability Could Have
