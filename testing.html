Testing

Testing Strategy

Testing combined automated Rust tests, frontend static checks, architectural boundary checks, and targeted manual end-to-end scenarios. This layered approach was necessary because SmolPC spans a shared local inference engine, a Tauri desktop shell, a Svelte frontend, and several host-application bridges. Low-level logic could be verified automatically, while workflows involving GIMP, Blender, and LibreOffice also required live scenario testing against real host applications.

Testing Approach

The test process moved from the inside out. Engine and client contract handling were verified first, followed by runtime selection, setup logic, provider behaviour, and launcher compatibility checks. These were then supported by integration and scenario testing across the unified application shell. Finally, prompt and user-oriented checks were used to review whether the assistants remained understandable and useful as an offline platform rather than just as isolated technical components.

Testing Tools

Automated verification used Rust's built-in test framework via cargo test, TypeScript and Svelte checks via npm run check, and architectural boundary checks via npm run boundary:check. Manual testing used Tauri development and packaged builds together with the relevant host applications installed, allowing real workflows to be exercised in GIMP, Blender, Writer, and Slides.

Evidence And Limitations

The repository already contains substantial automated coverage across the engine, the current Tauri backend under app/src-tauri, setup flows, connector crates, and host-app bridges. This page therefore focuses on the testing areas, the coverage achieved, and the behaviour observed across packaged builds, voice-resource checks, and hardware-aware runtime selection.

Unit Testing

Unit tests were used to lock down the most failure-prone logic: runtime configuration, API parsing, tool routing, host-integration heuristics, setup resource preparation, and error handling. This was especially important because small regressions at these boundaries can break the whole unified application even when the UI still appears to load correctly.

UT-LS1 Launcher and setup evaluate_engine_api_gate_accepts_matching_major, setup resource preparation tests in setup/provision.rs Checks version compatibility gates and confirms that required local resources can be staged before the user enters a mode. Covered

Integration Testing

Integration testing was essential because the platform is only useful when its parts work together: the shared engine must start correctly, the unified app must display the right state, each mode must route to the correct provider, and host-application bridges must behave predictably. Wherever possible, these boundaries were backed by automated tests; where external desktop applications were involved, live scenario testing was also required.

Test ID Components Tested Description Expected Outcome Status
IT1 Startup loading flow + setup state + engine readiness Launch the unified app, let the startup loading screen complete, refresh setup state, and confirm that engine readiness and mode availability are reflected in the main shell. The app should move from startup into the normal workspace only when the engine and setup state are coherent, with unavailable modes clearly flagged instead of failing silently. Covered
IT2 Mode registry + provider dispatch Switch between code, GIMP, Blender, and LibreOffice modes and confirm routing to the correct backend provider. Each mode should invoke the correct provider family without leaking behaviour across assistants. Covered
IT3 GIMP bridge + tool execution Prepare the local GIMP runtime, refresh available tools, execute an editing command, and test undo behaviour. The assistant should connect to the bridge, execute supported tools, and return a clear summary to the user. Covered with live scenarios
IT4 Blender bridge + local RAG Start the Blender bridge, read scene context, retrieve local documentation, and answer a scene-specific request. The assistant should combine bridge state and retrieved context without requiring cloud inference. Covered with live scenarios
IT5 LibreOffice runtime + planner/tool execution Run Writer or Slides requests that require structured tool calls and verify graceful handling of unsupported flows. Valid requests should execute through the office runtime, while unsupported routes should fail clearly and safely. Covered
IT6 Composer mic control + audio commands + engine voice endpoints Record a short prompt fragment, transcribe it locally into the composer, send the prompt, and request read-aloud on the response. The dictated text should appear in the chat input, and local playback should work or fail cleanly without breaking the text workflow. Covered in development builds

Prompt Testing

Prompt testing was used to check whether the assistants remained understandable and reliable in realistic user-facing scenarios. This mattered even for a local system because good offline deployment is not enough if prompts are interpreted inconsistently, mapped to the wrong tool, or explained poorly to non-expert users.

Test ID Mode Representative Prompt Expected Behaviour Status
PT-CH1 Code Mode "Explain this error and suggest how to fix it." The assistant should provide a clear coding explanation and next steps in a simple local chat workflow. Scenario defined and used during development
PT-VO1 Code Mode Voice Dictate a short coding question, then ask the app to read the answer aloud. Voice input should transcribe into the composer correctly, and the response should remain understandable when played back through the local read-aloud control. Scenario aligned with implemented UI flow
PT-GI1 GIMP "Blur the top half of the image." The assistant should choose an appropriate direct tool or fast path and report the edit clearly. Scenario aligned with automated heuristics
PT-BL1 Blender "How do I add a bevel modifier?" The assistant should retrieve relevant Blender context and give a scene-aware explanation. Scenario aligned with local RAG tests
PT-LO1 Writer / Slides "Add a heading called Hello." / "Add a slide titled Intro." The assistant should produce a valid structured tool call and return a useful summary of what changed. Scenario aligned with executor tests
PT-FB1 Cross-mode fallback Ambiguous or malformed tool-call output The system should fall back safely, summarise locally where possible, or ask for clarification instead of failing silently. Covered

Compatibility Testing

Compatibility testing focused on the environments and conditions that matter most for local deployment: packaged desktop behaviour, availability of host applications, and graceful fallback when a preferred runtime or dependency is missing. Because the project moved toward a unified locally bundled platform, compatibility is as much about setup and dependency handling as it is about raw operating-system support.

Platform or Condition Focus Expected Behaviour Current Position
Windows packaged build Primary deployment target for bundled local inference and host-app integration The installer and unified app should launch correctly, prepare local resources, and expose supported modes clearly. Primary validation target
Different local hardware profiles CPU, DirectML, and OpenVINO NPU runtime paths The platform should detect the best supported local runtime path and degrade gracefully if a faster path is unavailable. Covered in development across multiple runtime paths
Missing host applications or plugins Setup gating and user messaging Unsupported or incomplete host dependencies should produce a clear blocked or setup-required state rather than a crash. Covered by setup checks and manual scenarios
Missing microphone or unavailable audio output Voice feature degradation Voice controls should disable cleanly or report temporary unavailability while leaving the text-chat workflow unaffected. Implemented in UI and engine fallback flow

Performance and Stress Testing

Performance matters because SmolPC runs local models on varied hardware. The platform therefore tracks readiness state, runtime selection, memory pressure, streaming behaviour, and benchmark instrumentation. This section records the measurement areas that were monitored during development and the behaviours that matter most for real local deployment.

Test ID Area Metric Why It Matters Current Position
PF1 Engine startup Readiness time and startup robustness Slow or fragile startup undermines classroom or workshop usability even if generation quality is good. Measured during development
PF2 Streaming generation Time to first token, streaming continuity, response duration These metrics affect perceived responsiveness more than a single total latency number. Instrumented and observed during development
PF3 Runtime selection Correct backend choice for CPU, DirectML, or OpenVINO NPU The platform should use the best supported local path for the current machine rather than assume identical hardware everywhere. Selection logic covered
PF4 Memory pressure handling Warning and critical thresholds, model-switch guidance Protects weaker devices from becoming unstable during local generation. Heuristics covered by automated tests
PF5 Voice I/O Transcription turnaround, TTS availability, and added resource overhead Voice support should improve accessibility without making the main chat flow feel fragile or unexpectedly heavy. Validated in development builds

User Acceptance Testing

User acceptance testing was planned around representative target users rather than a single generic tester. This matched the product itself: the Code mode targets coding support, GIMP and Blender target creative workflows, and the LibreOffice modes target classroom and document-based tasks. Feedback was gathered informally and anonymously from representative users and peers across these workflows to check whether the product felt useful, understandable, and realistic to use as an offline assistant platform.

Tester Feedback

Code Mode Profile

Anonymous feedback on Code mode was positive overall. Testers found the chat flow easy to follow, valued the offline setup, and understood the assistant's answers well enough to identify sensible next steps in beginner-friendly coding tasks.

Writer / Slides Profile

Anonymous feedback suggested that the office workflows made simple document and slide actions clearer and faster to access, especially for users who preferred describing an action in plain language rather than searching through menus.

GIMP Profile

Anonymous feedback on GIMP was positive for straightforward edits and summaries, but it also confirmed that this remains the least consistent specialist mode and benefits most from clearer setup guidance and tightly scoped prompts.

Blender Profile

Anonymous feedback indicated that the Blender workflow felt more useful than a generic chatbot response because the assistant could combine scene-aware guidance with locally retrieved help content.

Project Partner Feedback

Anonymous stakeholder and partner-facing feedback was positive overall, particularly around the value of an offline-first assistant, the unified platform direction, and the practicality of a deployment story that does not depend on cloud accounts or a separate local model server.
