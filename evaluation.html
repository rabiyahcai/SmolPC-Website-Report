Evaluation

Summary of Achievements

This evaluation measures the delivered system against the functional requirements defined in the Requirements page. The strongest project outcome is not just one assistant mode in isolation, but the successful pivot toward a unified offline platform with a shared local engine, a common app shell, and assistant-specific workflows for coding, creative tools, and office tasks.

FR-M1 Natural-language text input across assistant modes Must have Achieved Text-based workflows are central to the unified app and all major assistant flows.
FR-M2 Unified desktop shell for Code mode, LibreOffice, GIMP, and Blender workflows Must have Achieved The platform moved beyond separate prototypes into a shared multi-mode desktop application.
FR-M3 Code mode supports coding questions, explanation, and debugging help Must have Achieved Code mode is the flagship workflow and provides a simple local coding assistant experience inside SmolPC 2.0.
FR-M4 LibreOffice assistant supports document and presentation workflows Must have Achieved Office workflows are supported within the unified platform, with room for further breadth and polish.

FR-M6 Blender assistant supports scene-aware 3D help and workflow guidance Must have Achieved Blender mode combines a local bridge with scene context and documentation retrieval.
FR-M7 Local on-device inference during normal use Must have Achieved The final architecture prioritises local inference rather than cloud-hosted AI services.
FR-M8 SmolPC 2.0 Home provides a unified launcher for supported tools Must have Achieved The shared shell and Home workflow give users one entry point into the platform instead of separate assistant launches.

Known Bugs

The core platform is usable, but several areas still deserve explicit tracking. These are not hidden failures; they are the main issues that would most improve reliability and deployment confidence if resolved before final submission.

ID Area Issue Priority Planned Direction
BUG-01 Setup and packaging Bundled prerequisites and host-application setup can still vary across machines, especially when local runtimes or tool plugins are not in the expected state. High Tighten packaged setup flows, improve validation, and expand final compatibility testing.
BUG-02 Host-application bridges GIMP, Blender, and LibreOffice workflows are inherently more fragile than Code mode because they depend on external application state, bridge availability, and version alignment. Medium Improve recovery messaging, bridge health checks, and plugin/runtime verification.
BUG-03 Benchmark and diagnostics path The benchmark path is still being migrated to the shared-engine contract, so performance tooling is not yet as polished as the main assistant workflows. Low Finish benchmark-contract migration and replace temporary legacy behaviour.
BUG-04 Complex multi-step prompts Long or ambiguous requests can still produce less predictable tool behaviour than narrow, single-purpose prompts. Medium Refine planner prompts, recovery strategies, and scenario testing for tool-heavy flows.
BUG-05 Voice I/O packaging and hardware assumptions Voice features depend on microphone access, audio output, and correctly bundled Whisper and TTS resources, so environment-specific issues are still more likely here than in plain text chat. Medium Complete final packaged validation for voice resources and improve recovery messaging when audio features are unavailable.

Critical Evaluation

The project should be judged not only by feature count, but by whether the final direction solves the real deployment problem it set out to address. In that sense, the most important success was the shift from disconnected or harder-to-deploy local prototypes toward a more coherent shared-engine platform that better fits an educational offline use case.

Functionality

Functionally, the system now covers all major intended tool families: coding assistance, office workflows, image editing, and Blender support. It also now includes optional local voice input and read-aloud in the unified app. The unified shell is a major improvement over earlier separate prototypes because it gives the project a single identity and a common interaction model. The trade-off is that some specialist flows still feel narrower and more constrained than the strongest core path, which remains Code mode.

Code Mode

Code mode succeeds as a simple local coding assistant and is the clearest expression of the platform's offline-first goal. Its main weakness is that it is still closer to a detached chatbot than to a deeply embedded coding workflow, which leaves room for richer context and stronger task integration in future work. The addition of local voice input and read-aloud also strengthens the accessibility story of the wider SmolPC 2.0 platform.

LibreOffice Assistant

The office modes are valuable because they preserve the original educational productivity motivation of SmolPC, but they also highlight the difficulty of bridging natural language into structured host-application actions. The current outcome is useful and credible, yet still a good candidate for more tool breadth and refinement.

GIMP Assistant

GIMP mode shows the value of tool-aware editing rather than generic chat. It is one of the clearest examples of why app-specific bridges matter, although its usefulness still depends on reliable setup and host-state alignment.

Blender Assistant

Blender mode is particularly strong conceptually because it combines scene awareness with local retrieval. It demonstrates how a specialist assistant can do more than answer generic questions, though the surrounding tooling remains more complex than a pure chat mode.

Stability

Stability improved significantly once common responsibilities were moved into the shared engine and unified shell. Automated tests now cover many of the failure-prone boundaries, including runtime selection, engine-state mapping, setup preparation, provider logic, and compatibility gates. The main remaining stability risks come from the parts of the system that depend on external desktop applications and their local plugin environments.

Shared Engine and App Shell

Centralising startup, runtime choice, and model orchestration made the platform easier to reason about and test. Keeping speech-to-text inside the engine while isolating text-to-speech in a detached sidecar also reduces the blast radius of voice-specific failures. This architectural consolidation is one of the strongest technical decisions in the whole project.

Host-Integrated Modes

GIMP, Blender, and LibreOffice remain inherently more fragile than Code mode because they rely on bridge processes, external applications, and setup assumptions. They are therefore stable enough to demonstrate the concept, but still the first place where environment-specific issues are likely to appear.

Efficiency

Efficiency should be understood in context: this is a local-first AI platform, so acceptable performance depends on startup behaviour, streaming responsiveness, memory pressure handling, and choosing the best supported runtime for the current device. The architecture is well aligned with that goal, but final performance reporting still needs one more benchmark pass now that the shared-engine design has stabilised.

Runtime Selection

The move to hardware-aware runtime selection is a major improvement over a fixed local stack. It allows the system to aim for the best supported local performance rather than assuming one narrow hardware profile.

Model Trade-Offs

The platform demonstrates a realistic trade-off between model quality, responsiveness, and device capability. Smaller models can help with fallback behaviour, while stronger baseline models improve reliability across the wider set of assistant modes.

Compatibility

Compatibility remains strongest on the Windows-first packaged path that shaped most of the runtime and host-application work. The shared engine improves consistency across different local hardware by choosing between CPU, DirectML, and OpenVINO NPU where supported, while host-based modes still depend on the relevant external application being installed and reachable. Voice features add another compatibility layer because microphone access, audio output, and bundled voice resources must all be present, but the current design degrades these features more safely than a monolithic in-process approach would.

Maintainability

Maintainability improved substantially after the architecture pivot. Separating responsibilities across the shared engine, the app shell, and the mode-provider layer makes the system easier to extend than the earlier prototype path. The remaining challenge is coordination: because several layers interact, changes to contracts or setup behaviour still need careful documentation and regression testing.

UI/UX

The unified interface is a strong usability improvement because it gives the project a single consistent entry point instead of multiple disconnected tools. Code mode currently offers the clearest and simplest user journey, and the added mic and read-aloud controls strengthen its accessibility story. The more specialised modes benefit from the same shell but still need clearer setup guidance, richer onboarding, and more polish around error recovery and workflow explanation.

Project Management

Project management was shaped by several major technical pivots. Although this increased complexity, the pivots were not arbitrary: they were responses to real deployment, packaging, and hardware constraints. In hindsight, that adaptability was a strength, because it led to a more coherent and deployable final direction. The main cost was that some polish work had to happen later than ideal because foundational architecture decisions were still evolving during the implementation period.

Future Work

The next stage of work should focus less on adding brand-new ideas and more on turning the current platform into a more polished, better-measured, and easier-to-deploy system. The architecture is now strong enough that future effort can concentrate on reliability, workflow depth, and accessibility.

Short-Term Improvements

Code Mode

Move beyond a detached chat-box workflow by improving context injection, project awareness, and support for richer coding interactions while preserving the simple offline model.

LibreOffice Assistant

Expand the breadth and robustness of office workflows, especially around structured tool usage, clearer summaries of actions taken, and smoother recovery from missing prerequisites.

GIMP Assistant

Reduce setup friction, broaden the range of reliable image-editing operations, and improve the clarity of edit summaries and undo behaviour.

Blender Assistant

Deepen scene introspection, strengthen documentation retrieval quality, and improve the flow between natural-language guidance and concrete Blender actions.

Long-Term Vision

Additional Assistants

Deeper Integration

Over time, the platform could move from separate assistant modes toward stronger cross-tool continuity, shared context, and a more seamless workflow between coding, documents, graphics, and 3D tasks.

Performance, Accessibility, and Deployment

A future release should complete the benchmark migration, expand hardware validation, add more accessibility options, broaden voice support beyond the flagship chat flow, and improve packaging so that local deployment is as smooth as the underlying architecture now promises.
